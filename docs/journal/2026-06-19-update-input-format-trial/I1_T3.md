# 試験用 large issue (update 実装比較のための原本)

## 概要

これは local-issue:update スキルの入力形式 (= 親 AI から subagent への橋渡し)
を比較するための試験用 issue ファイル。実際の large issue (= 議論を追記しながら
成熟させる用途、docs-structure 規約で OK とされている) を模した内容と分量を
持つ。3 つの異なる入力形式 (自然文 / Edit ペア配列 / section 指定) で同じ修正を
実施し、使い勝手・正確性・副作用を比較する。

## 背景

local-issue は claude-rules-personal の docs-knowledge-flow が定義する issue 運用
を発展させた独立プラグイン (DR-0002)。close 時に削除でなく archive へ物理移動
する設計、status 7 値 (idea / open / wip / blocked / pending-sublimation /
discarded / resolved) と category 6 値 (idea / bug / request / design / task /
tech-memo) の 2 軸分離、時系列メタの full ISO8601 記録などが特徴。

本試験は local-issue 自身の改善に向けた dogfooding。kawaz の指示 (2026-06-19
nandakke 設計セッション中) で開始された。

## 受け入れ条件

- [ ] 3 つの入力形式で同じ修正タスクを実施できること
- [ ] 修正の正確性 (= 期待通りの結果) を比較できること
- [ ] 入力構築の手間 (= 親 AI が prompt を組み立てるコスト) を計測できること
- [ ] 副作用 (= 意図しない場所への変更) の有無を検証できること
- [ ] 主観的な使い勝手の評価を残せること

## 検討事項 (改訂版)

local-issue:update の入力形式は subagent への橋渡しが本質。親 AI のコスト
(prompt 構築) と subagent のコスト (解釈精度) のトレードオフを最小化する設計が
最終的な使い勝手を決める。

本試験では小〜中規模の修正タスクで 3 形式を比較するが、大規模な場合の挙動は
別途検証が必要。

## 解決時の記録先

- 試験結果は `docs/journal/2026-06-19-update-input-format-trial/` に集約
- 結論を踏まえて local-issue リポに improvement issue を起票
- 本試験用 issue は実 issue ではないので archive 不要

## 試行メモ (subagent)

- ツール呼出数: Read × 2 (source + copy確認)、Bash × 2 (ls確認 + cp)、Edit × 1 (section置換)、Edit × 1 (末尾追記) = 計 6 呼出
- 解釈に迷った点: section の終端境界。「次の見出し「## 解決時の記録先」の直前まで」と明示されていたため境界判定は容易だった。sub-section (### 行) が複数あるにも関わらず見出しレベルを問わずまとめて置換対象と指定されていたので、old_string を「## 検討事項」から「## 解決時の記録先」の直前まで丸ごとマッチさせる方針で対処した。
- 使いやすさ: 2/3。自然文で「見出しから次の見出しの直前まで」という範囲指定は伝わりやすい。ただし old_string に渡す文字列を subagent が自力で組み立てる必要があり、長い section だと old_string が大きくなる。section 名だけで境界を指定できる専用構文があればより効率的。
- 所感: 今回の入力は範囲が明示的で誤解の余地が少なかった。「中間の sub-section も含む」という補足も効いており、解釈ぶれはなかった。section 名のみの指定 (構造依存) と比較すると、見出しテキストが変化した場合に自然文指定は追随しやすい利点がある。
