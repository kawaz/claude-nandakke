# 試験用 large issue (update 実装比較のための原本)

## 概要

これは local-issue:update スキルへの入力形式 (= 親 AI から subagent への橋渡し)
を比較するための試験用 issue ファイル。実際の large issue (= 議論を追記しながら
成熟させる用途、docs-structure 規約で OK とされている) を模した内容と分量を
持つ。3 つの異なる入力形式 (自然文 / Edit ペア配列 / section 指定) で同じ修正を
実施し、使い勝手・正確性・副作用を比較する。

## 背景

local-issue は claude-rules-personal の docs-knowledge-flow が定義する issue 運用
を発展させた独立プラグイン (DR-0002)。close 時に削除でなく archive へ物理移動
する設計、status 7 値 (idea / open / wip / blocked / pending-sublimation /
discarded / resolved) と category 6 値 (idea / bug / request / design / task /
tech-memo) の 2 軸分離、時系列メタの full ISO8601 記録などが特徴。

本試験は local-issue 自身の改善に向けた dogfooding。kawaz の指示 (2026-06-19
nandakke 設計セッション中) で開始された。

## 受け入れ条件

- [ ] 3 つの入力形式で同じ修正タスクを実施できること
- [ ] 修正の正確性 (= 期待通りの結果) を比較できること
- [ ] 入力構築の手間 (= 親 AI が prompt を組み立てるコスト) を計測できること
- [ ] 副作用 (= 意図しない場所への変更) の有無を検証できること
- [ ] 主観的な使い勝手の評価を残せること

## 検討事項

### 入力形式の比較軸

入力形式は親 AI (Opus/Sonnet 等) と subagent (Sonnet) の間の通信プロトコル。
親 AI 側のコスト (prompt 構築の手間と token 量) と subagent 側のコスト (解釈の
精度と Edit 操作の安定性) のトレードオフが存在する。

中規模 issue (1-5 KB) では自然文で十分機能するが、大規模 issue (5 KB 超、議論
追記型) では自然文の embed が肥大化し、subagent の解釈ぶれリスクが上がる。

### 期待される使い分け

- 1 行 typo: 自然文で十分、Edit ペア配列はオーバーキル
- 数箇所の置換: Edit ペア配列が直感的
- section 全体書き換え: section 指定が自然
- 複合操作: section 指定 + 内部 Edit ペアの階層構造

### 棚上げ事項

- patch (unified diff) 形式は親 AI が diff を組み立てる癖がないため棚上げ
- 行範囲指定 (`--lines N..M`) は markdown structure と相性が悪いため read 側のみで採用

## 解決時の記録先

- 試験結果は `docs/journal/2026-06-19-update-input-format-trial/` に集約
- 結論を踏まえて local-issue リポに improvement issue を起票
- 本試験用 issue は実 issue ではないので archive 不要

## 試行メモ (subagent)

試行 I3_T1: section 指定形式 × typo 1 文字修正。

**操作**: `section: "## 概要"` + `new_content: "..."` を受け取り、section 全体を Edit で置換した。

**typo 修正に section 全体を渡す違和感について**

変更箇所は「スキルの入力形式」→「スキルへの入力形式」の 1 文字 (助詞) だけだったが、
section 指定形式では section ブロック全体 (6 行) を new_content として渡す必要があった。
この不釣り合い感は大きい。具体的には:

- **親 AI 側の負担が高い**: 1 文字直すだけなのに、正確な section 全文をコピーして
  助詞だけ書き換えた版を prompt に embed しなければならない。section が長いほどこの
  コストが指数的に上がる。
- **転記ミスのリスク**: section 全体を再入力 / コピーする過程で、意図しない別の変更が
  混入しやすい。今回は短い section だったので問題ないが、長い section では副作用の
  検出が難しくなる。
- **「section 指定」の本来の強み**: section 全体を書き換えたい場合 (構成見直し、大幅な
  加筆) には向いている。しかし 1 行 typo に使うのはオーバーキル。設計文書の想定する
  「section 全体書き換え」ユースケースには合っていない。

**結論**: section 指定形式は typo 修正には不適。1 文字変更には Edit ペア配列 (old/new の
差分のみ渡す) か自然文の方が親 AI のコストが低く、転記ミスのリスクも少ない。
section 指定形式は「section を丸ごと新しい内容で置き換えたい」場合に限定すべき。
