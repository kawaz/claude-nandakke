## 4. 実装案ごとの強み・弱み

### I1: 自然文

強み:
- 親 AI のコストが低い: 意図を短く書くだけで伝わる
- 横断的な指示が書ける: 「本文全体から X を Y に置換して」で section 境界を意識しなくてよい
- 変更の粒度を問わない: typo から section 全体書き換えまで同じ記述スタイルで対応

弱み:
- subagent の解釈依存: 「どこを」の特定を subagent に委ねるため、複雑な指示ではぶれが生じうる
- 全置換手段が Write になりがち: ファイルが大きくなると Write の全体再書き込みはコスト増
- Write リグレッションリスク: 全文再書き込み時に subagent の転記ミスが入りうる

### I2: Edit ペア配列

強み:
- subagent 側がほぼ機械的: 配列を Edit ツール呼び出しへの変換だけで解釈余地がない
- 複数箇所への点的修正に強い: 複数の独立した変更箇所を一度に指定できる
- 副作用がない: 指定した箇所以外は絶対に変わらない

弱み:
- 親 AI のコストが高い: old_string を正確に抜き出す必要があり、1 文字の誤りでマッチ失敗
- 指定漏れが即ミスになる: 全置換タスクで指定箇所が不完全なら部分置換で終わる (今回の I2_T2 の失敗)
- section 全体書き換えには不向き: old_string が長くなり prompt が肥大化する

### I3: section 指定

強み:
- section 全体書き換えに最もフィット: section 名 + new_content で意図が最もシンプル
- 境界の曖昧さがない: 次の ## までが自動的に範囲になるため一貫した解釈になる
- 副作用リスクが構造上低い: 隣接 section を巻き込む誤操作が起きにくい

弱み:
- 横断的変更に弱い: フレーズが複数 section にまたがると section 数に比例してコスト増
- typo 修正には不釣り合い: 1 文字直すために section 全文を embed する必要がある
- new_content の転記ミスリスク: 長い section を全文再入力する過程で意図しない差分が混入しやすい

## 5. 結論: 使い分けマトリクス

| タスクの性質 | 推奨形式 | 理由 |
|-------------|---------|------|
| 1 箇所の typo / 1 行修正 | 自然文 (I1) | 最も手軽。Edit ペア配列は old_string 抽出コストが割に合わない |
| 数箇所の点的置換 (section をまたぐ、かつ箇所が確定している) | Edit ペア配列 (I2) | 横断的かつ正確。subagent の判断余地がない |
| 1 section 全体の書き換え | section 指定 (I3) | 境界が明確、new_content をそのまま渡せる |
| section をまたぐフレーズ全置換 (箇所が不確定) | 自然文 (I1) | I3 は構造上 1 section しかカバーできない。I2 は全箇所の確認が前提 |
| 複合操作 (section 書き換え + 点的 typo 修正) | section 指定 (I3) + Edit ペア配列 (I2) の組み合わせ | 各形式の得意領域を組み合わせる |

形式選択の判断軸: 「変更の局所性 (= 何 section にまたがるか)」と「変更の粒度 (= 文字・行・section)」の 2 軸で決める。

## 6. local-issue への improvement issue 候補

### 6-1. 全置換タスクにおける I2/I3 の指定漏れ対策

問題: Edit ペア配列と section 指定は「指定した範囲しか処理しない」ため、横断的な全置換タスクで必然的に取りこぼしが発生する。
提案: SKILL.md に「フレーズが複数 section にまたがる場合は自然文または Edit ペア配列 (全箇所指定) を使用する」旨を明記する。

### 6-2. I1 (自然文) の Write 依存問題

問題: 自然文で「複数箇所を置換して」と指示すると、subagent が Edit の replace_all パラメータを使うか Write で全文再書き込みするかを自己判断する。Write は大きいファイルでコスト増、かつ転記ミスが入りうる。
提案: SKILL.md に「全置換は Edit ツールの replace_all パラメータを優先し、Write での全体再書き込みは避ける」を明記する。

### 6-3. section 指定形式の section キー仕様の明文化

問題: section ヘッダを変更しながら内容も変える場合、section キーが変更前ヘッダを指すのか変更後ヘッダを指すのかが暗黙になっている。
提案: SKILL.md に「section キーは変更前のヘッダ文字列 (完全一致) を指定する」と明記する。

### 6-4. ツール呼出数の記録標準化

問題: 今回の試行メモで「ツール呼出数」の記録フォーマットが試行によってまちまちだった (合計数のみ / 内訳付き / 記録なし)。
提案: 試行テンプレートに「ツール呼出数: Read × N, Edit × N, Write × N (合計 N 回)」のフォーマットを固定する。

### 6-5. 全置換前の出現箇所確認を推奨ステップとして追加

問題: 「親 AI」5 箇所のうち 3 箇所が起票者に把握されておらず、I2/I3 では指定漏れが発生した。
提案: update skill の呼び出し前に「対象フレーズの出現箇所をまず grep で確認する」ステップを推奨する旨を SKILL.md に追記する。親 AI が grep を実行してから edit 指示を構築するフローが望ましい。
