## 迷った点

- **`-t T` は思考 (Thinking) のみを取得する**条件。AI の内部推論が連続して流れるため、「何を決めたか」の結論は比較的読みやすい。一方で「実際に何を編集したか」がタイムラインには現れない (file-ops で補完が必要)。
- **思考が英語と日本語で混在している**。同一セッション内で言語が切り替わる箇所があり、どちらが「本流の推論」かを判断しにくい。日本語側の方が実際の作業文脈に近い傾向があるが確証はない。
- **turn 13 が非常に長い** (screen snapshot 実装全体)。思考エントリが 10+ 件連続して 1 turn に圧縮されており、途中で方針転換があったかどうかの追跡が難しい。
- **codex stop-time review の指摘内容**が思考経由でのみ参照できる。codex review 自体の出力は -t T では見えないため、「何が指摘されたか」は AI の要約に依存することになる。
- **ultraplan ワークフローの途中切断**。output file を読もうとしたが、サブエージェント transcript 読み禁止ルールの適用範囲に迷っている様子が turn 16 に記録されている。最終的に読まずに codex 結果と turn 17 で補完している。

```bash
# 範囲分割パターン (今回使用)
timeline ... 1..6     # → persisted-output に逃げる (42KB)
timeline ... 7..12    # → 直接読める (~400行)
timeline ... 13..15   # → 直接読める (~500行)
timeline ... 16..23   # → 直接読める (~600行)
```

### 観察: persisted-output の再帰問題

- turn 1..6 の範囲でも 42KB を超えて persisted-output に保存される
- 保存先の `.txt` を `cat` しても再び persisted-output でラップされ読めない
- 対策: 出力を `grep -v "^Output too large" | grep -v "^Full output saved" | grep -v "^Preview"` でフィルタ後 `head -N` で取得すると回避できる
- ただし 1..6 でも大きすぎる → 1..4 / 5..6 のように更に細かく分割が必要な場合がある

### file-ops の使い方

```bash
claude-session-analysis file-ops -d 1 269559b3-fb71-4a89-bae6-95940a4aa65e
# -d 1 で depth=1 の操作一覧 (tool / path / timestamp / turn が JSONL で出力)
# timeline -t T と組み合わせることで「思考が何を読み/編集したか」を補完できる
```

### -t T の特性まとめ (今回の観察から)

| 項目 | 観察 |
|---|---|
| 出力内容 | AI の思考 (Thinking) のみ。ユーザ発言・ツール結果・最終応答は除外 |
| 言語 | 英語・日本語が混在 (ターンによって異なる) |
| 長い turn | 複数の思考エントリが連続して 1 turn に圧縮される |
| 情報密度 | 「なぜそう判断したか」「何に迷ったか」が非常によく読める |
| 見えないもの | 実際のファイル差分・ツール出力・ユーザーの反応 |
| file-ops 補完 | 「何を編集したか」は file-ops -d 1 で別途補完が必要 |
