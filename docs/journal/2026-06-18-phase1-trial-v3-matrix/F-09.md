## 手順メモ (条件 F で分かったこと)

- **width 9999 の効果**: T (thinking) と R (response) の両方が 1 行も折り返さず出力された。2.4MB セッションでも全体を 1 パスで読める。
- **分割が必要なポイント**: 2400 行超えで `tail -n +N | head -M` 方式での範囲分割が必要。今回は 5 分割 (0-200 / 200-500 / 500-900 / 900-1300 / 1300-1700 / 1700-2100 / 2100-2400 / 2400-) で完走。実際には 2600 行以降は空だった。
- **T と R の情報密度差**: T (thinking) はエージェントの判断過程・迷い・却下候補を含む。R (response) は確定した行動と結果のみ。Phase 1 の「索引」目的には R だけで十分だが、「なぜその決定をしたか」を追うには T が必須。
- **file-ops `-d 1` の用途**: セッション内の Read/Edit/Write 操作が時系列で追える。どのファイルをどの turn で触ったかが一覧化され、「この修正は turn 13 の連続 Edit」のような追跡が可能。ただし内容は見えないのでタイムラインと組み合わせて使う。
- **md=source の利点**: マークダウンの見出し・テーブル・コードブロックが崩れずに出力される。特にテーブルが多い R (response) では render より source の方が情報損失が少ない。
