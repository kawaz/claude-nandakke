## この条件で見えなかったこと

**R 型 (Claude のテキスト応答) + md=source で読む制約で以下が得られなかった。**

1. **ユーザの発言原文・指示の詳細 (U 型)**
   - R 型の応答から「kawaz がどう答えたか」を推測できる場面は多い (「賛成。」「了解。」「軌道修正。〜」等の応答から前 kawaz 発言を逆算できる) が、Q1/Q2 への回答や設計判断の指示がどのような言葉だったかは直接見えない。例えば tell 廃止方針の転換は turn 10 R3b7cb27d から分かるが、kawaz の具体的な言葉は不明。

2. **実装の試行錯誤・テスト失敗の経路 (T 型 / B 型)**
   - turn 20 の Non-Stop モードで 21 commit が一気に land されているが、各実装ステップでどのようなエラーが出て何度やり直したかは R 型の報告 (「stage A 全 pass」「stage C 9 件 pass」等) からしか分からない。B 型があれば実際の test 出力や中間エラーを追える。
   - scope-hash.ts の `/private/tmp` symlink 問題 (R2fe0c287) は R 型に「macOS の symlink で realpath の扱いがズレていた」とあるが、どのパスを試してどう変えたかは B 型なしでは追跡不可。

3. **hyoui の実機操作 (turn 6: 練習セッション) の詳細**
   - 子 claude を hyoui で起動してメッセージを送る練習が turn 6 で行われたが、R 型には「観察できたこと / 気になる挙動」の要約しかなく、実際の hyoui コマンドの出力・ANSI 文字列の生データ・画面状態は B 型・T 型なしでは見えない。

4. **複数の DR 草稿の中間状態**
   - DR-0015 が 3 commit に分かれて改訂されている (初版 → label 統一 → 4 軸化 → 確定版) ことは R 型の「DR-0015 改訂完了」系報告で追えるが、各段階の実際のファイル内容は R 型からは分からない。file-ops で Write パスだけ分かる。

5. **claudeのskill/command model 切替事故の実際の画面 (turn 34〜57)**
   - turn 34 に `Prompt is too long`、turn 57 に `Prompt is too long` が R 型に出ているが、これはセッション自体が context limit で中断していた証拠。この一連の事故 (haiku で limit reached → resume → sonnet で API Error → resume...) の経緯は R 型だけでは断片的にしか見えず、U 型・T 型がないと再現が難しい。

6. **他セッションへのメッセージ内容**

## 解釈に迷った点

1. **2 日間大規模セッションの「相」の把握難易度**
   - 78 turn 、timeline が 2700 行超で、R 型のみを読んで全 topic を把握するのに timeline を 9 回に分割して取得した。索引の「最終言及」列に turn 番号を記録したが、一度 spoken → recorded → implemented と遷移した topic がどの段階でどこに記録されたかを追跡するのが R 型のみでは重い。T 型や B 型があればファイルが実際に書かれたかどうかで即判定できるが、R 型は「完了しました」の宣言しか見えない。

2. **turn 34〜57 の Prompt is too long 連発区間**
   - この区間は R 型の出力が `Prompt is too long` や `API Error: Usage credits required` のみで、Claude の通常応答が出ていない。セッションが context limit に何度か当たっていたことは分かるが、どこで resume されて文脈がどこまで保持されているか R 型だけでは不明確。「索引としてこの区間の topic を記録できるか」が難しい箇所だった。

3. **Non-Stop モード中の自律判断の記録粒度**
   - turn 20 の Non-Stop モードで 21 commit が自律 land されているが、R 型の報告は「stage A 完了 → stage B 完了 → ...」という粒度。何が判断されてどの順に作られたかは R 型で追えるが、「なぜ scope-hash より前に whoami を先に作ったのか」等の細かな判断経緯は R 型には出てこない。

4. **`@parcel/watcher` 不採用の根拠の解釈**
   - 当初 subagent 報告で「Bun fs.watch は broken (#3657)」として `@parcel/watcher` を採用予定だったが、実機検証 (turn 9) で「macOS Bun 1.3.13 では再現せず」と判定が覆った。R 型には「subagent の予測を覆した」という文章があるため「事実」扱いにしたが、Linux CI での検証結果が明示されていないことが気になった (findings ファイル参照を推奨の next にした)。

5. **DR 起票後の claude-plugin-reference への反映 (push 据え置き) の扱い**
   - turn 65〜67 で claude-plugin-reference への知見反映を working copy に書いたが、kawaz 指示で push 据え置きになった。その後 jj op restore で一度消え、restore で戻った。この「working copy に存在するが push されていない」状態が R 型では分かりにくく、最終的に「推定」確信・push 未の旨を next に記した。
