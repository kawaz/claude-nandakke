## この条件で見えなかったこと

**R 型 (Claude のテキスト応答) + md=source で読む制約で以下の情報が得られなかった。**

1. **ユーザの実際の発言原文** (U 型)
   - R 型には「了解。方針確定：」「確認する。」「実装する。」「了解。実装する。」のように Claude の応答からユーザ発言が推測できる部分もあるが、ユーザが何をどんな言葉で指示したか (U 型) は直接見えない。
   - 例: turn 5 の `-m N` 機能要求がどのような言葉で出たかは R 型からは「了解。実装する。」としか読めず、要件の詳細は不明。

2. **ツール実行の実結果** (Bash 出力 / テスト生ログ)
   - R 型に「CI 全通過。」「型チェックと動作確認：」「複数 range の OR 結合が動作している」とあるが、実際の test 出力やコマンド出力は載っていない。エラーがあったかどうか、何を叩いたかは B 型がないと分からない。

3. **試行錯誤の経路** (T 型 / B 型)
   - R 型の多くは「次に〜する。」「〜を確認して反映。」という短い宣言か、最終的な変更まとめレポートで構成される。何度試みたか、どのアプローチを途中で捨てたか、test が何回失敗したかは T 型・B 型なしには読めない。

4. **ファイルの実際の差分・最終状態**
   - どのファイルのどの行が変わったかは file-ops から「Edit した」事実しか分からない。R 型の変更まとめ (「`search.ts` の cwd 最終化」等) は変更の要約に過ぎず、実際のコード片は見えない。

5. **cwd ドロップ判断の詳細経路 (turn 2→3 のギャップ)**
   - R32b9b5fd (turn 2) は `/cd` による cwd 変更の問題観察と方針の検討で、二択 (last 単一 vs last + firstCwd 併記) を提示している。しかし次の R 型 (R00e1dc7a turn 3) はすでに「方針確定」から始まり、ユーザが「last 単一で OK」と答えた経緯が R 型だけでは確認できない (U 型がないと判断者が分からない)。

## 解釈に迷った点

1. **R 型のみで「相」を判断する難しさ**
   - R 型の変更まとめに「CI 全通過。push (bump 含む) は指示待ち。」という表現が turn 3, 4, 5 すべてで繰り返されている。これを「implemented」とすべきか「push 待ちなので recorded 止まり」とすべきか迷った。今回はセッション内で実装・テスト通過までを confirmed として「implemented」に分類したが、リポへの push が確認できていないため確信は「推定」にとどめた。

2. **R 型の末尾まとめが詳しすぎて逆にポインタとして冗長**
   - R 型 (md=source) の各ターン末尾には「## 変更まとめ」「## 結果報告」等の詳細が記載されており、内容自体は豊富。しかし索引は「ポインタ + 相」であればよく、これだけ詳しく読む必要があるのかは疑問。A 条件 (T 型) と比べると「内容量多いが索引に必要な相・経緯の判断には過不足がある」という印象。

3. **turn 2 (R32b9b5fd) の位置づけ**
   - turn 2 の R 型は、問題観察 + 方針提案で終わっており「ユーザへの提案・質問」に当たる。相としては「spoken (検討中)」だが、turn 3 で「了解。方針確定」となり即 implemented に至っている。ターン内でのステート遷移を索引に記録する粒度 (「turn 2 で spoken、turn 3 で implemented」) をどう表現するか迷い、最終的に「最後に言及した turn」を採用した。

4. **md=source が索引目的で有用かどうか**
   - `--md=source` にすると Markdown が生テキストで出力される (レンダリングなし)。このセッションの R 型には箇条書きや見出しが多用されており、source のままでも構造は読み取れた。ただし「索引として何が起きたか」を把握する目的では、R 型全文を読む必要はなく要約だけで十分な場面が多かった。md=source よりも md=render (またはなし) の方が実際の文字数は同じだが視覚的な把握が速くなる可能性がある。
