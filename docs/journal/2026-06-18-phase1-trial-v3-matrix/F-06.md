## 見えなかった

**timeline が 1 ターン分しか出力されない** — セッション全体は 30 行 JSONL だが、`-t TR` が拾える T エントリは 1 箇所 ("Looking at the diff...") のみで、最終的な review 結果テキストは存在しない (セッションが途中で落ちているため)。

1. **reviewer が実際に何を発見したか** (セキュリティ指摘の内容)
   - セッションが Grep の rg ENOENT でクラッシュして終了しているため、review findings は一切生成されていない。T エントリ 1 行 ("I need to analyze the security implications of the new iMessage notification functionality") しか残っておらず、何が問題だったかの結論は存在しない。
   - diff (queue-operation の content) 自体は残っているので、diff の内容は参照可能。ただし "review した結果" は存在しない。

2. **最初の `/Users/ksh/bin/...` パス誤りの根拠**
   - reviewer は最初に `/Users/ksh/bin/classroom-monitor` を試みている (line 8)。なぜ `/Users/kawaz/...` ではなく `/Users/ksh/...` を試みたのかは T エントリに記録がなく、queue-operation の content にも絶対パスの記載はない。セッション開始時に補完した推測パスが誤りだったと見られるが、その推論過程は本条件では読めない。

3. **rg 未インストールの事前認識**
   - Grep ツールが `rg` を内部コマンドとして使っていること、それがこの実行環境 (worktree 内) では未インストールだったことは、クラッシュして初めて判明する。T エントリには「grep で関数を探す」という意図はあるが、rg が使えるかどうかの確認判断は存在しない。

4. **timeline コマンドが T/R 1 エントリしか返さない理由**

## 迷った

1. **queue-operation の diff は "spoken" か "recorded" か**
   - diff 全文 (diff + 依頼文) が queue-operation の content として JSONL に永続化されている。これは「ファイルに落ちた」という意味では "recorded" に相当するが、nandakke の索引スキーマが想定する "recorded" はプロジェクト知識 (DR/findings/journal 等) に相当するものであり、reviewer への一時的な入力 (queue-operation) は通常の "recorded" と同一視しにくい。今回は "spoken" で記録した。

2. **セッションがクラッシュした場合の索引の意義**
   - 結果が存在しないセッションを索引化することに意味はあるか — 「review が完了していない」という事実を "spoken / 推定" で明示することで、次回触れたときに「継続が必要」と気づける。索引なしだと「review したかどうか」すら分からなくなる。今回は索引を作る意義があると判断した。

3. **timeline の `command_computed` で `1 R5b0afe24..1 R5b0afe24` と同一 range になっている理由**
   - このセッションは開始〜終了まで同一 root message に属する (branch なし、single path) ため、range が 1 点に縮退している。複数のブランチや compact 履歴があるセッションとは構造が異なる。

**timeline が 1 エントリしか返さない観察**:
`command_computed:` に `1 R5b0afe24..1 R5b0afe24` とある — 同一 range で、T/R 型の assistant エントリが line 21 の text エントリ 1 箇所しかないため。クラッシュで終了したセッションでは timeline の情報量が極端に少ない。

**このセッションに `-t TR --md=source` を適用した効果**:
