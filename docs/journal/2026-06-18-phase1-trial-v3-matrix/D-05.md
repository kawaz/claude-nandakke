## 迷った判断

### T のみで R なしの影響

-t T は thinking だけなので、Claude がユーザに何を最終報告したかが追えない。
R があれば「要約」として最終出力が取れるが、T だけでは推論過程しかない。
本セッションでは T が詳細で最終判断が読めたが、「ユーザへの返答文」は復元できなかった。

### 長いターン (turn 8) の扱い

turn 8 は複数の問題 (workspace 破損 / version ズレ / CI 失敗 × 2 / justfile 修正) を
1 ターンで処理しており、T が非常に長かった。`--width 9999` により折り返しなしで 1 行に
なったため、行数は少なくてもトークンは多い。索引用途では width を絞って短縮する余地がある。

### Bash 操作が file-ops に現れない

`rm` (issue 削除) と sed / Python (justfile 修正) は Bash ツール経由のため、
file-ops には Edit/Write だけが記録され、Bash 操作は出ない。
T に「Bash で実行」と書いてあるため T から追えるが、file-ops だけを見ると
「何も変えていないのに push された」ように見える盲点になりうる。

### settings.local.json の発見

T19 で「settings.local.json が gitignore 外でも jj では untracked になる」という
発見があった。T のみのためこの判断が最終的に何かの変更に落ちたかどうか不明
(file-ops に Write がない = ファイルは作られていない)。
finding として残す価値がある発見だが、R なしでは「提案にとどまったか実行したか」が追えない。

### 気づいた挙動

- `--md=source` は Markdown をレンダリングせずソース文字列として出力する。
  T の中の Markdown 記法 (表・コードブロック等) がそのまま読める。
- `-t T` のみのため出力行数は少なく、width 9999 でも Bash で扱いやすいサイズに収まった。
- `command_computed` に `--width 0` と表示される (9999 は no-limit 扱い)。
- file-ops で Bash ツール操作が記録されない点は、T と突き合わせる際の注意点になる。
- R がないため Claude の最終応答が追えず、「セッションの終わりがどこか」が判断しにくい。
  最後の T エントリが turn 19 だが、その後さらに会話があったかどうかが T だけでは不明。
