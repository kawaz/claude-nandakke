## 解釈に迷った点

1. **R 型と T 型の境界**
   - `Rd66bfa45` が T 型相当の長い分析文であり、`R66f41d28` が 1 段落の結論文。通常 T 型はエントリ ID が `T...` で始まるが、本セッションでは `-t TR` の R 型として `Rd66bfa45` が出てきた。`R` 始まりのエントリが思考ブロックの一部なのかツール結果なのかが不明。ただし内容から「Claude の分析テキスト = 思考ブロック」と判断した。

2. **1 ターンセッションにおける条件 F の情報量評価**
   - セッション自体が 1 ターンのため、TR 条件で出力される情報量は 31 行と極小だった。`--md=source` の効果もほぼゼロ（Markdown 構造がないテキストのため）。条件 F の評価として「セッションが長いほど T/R の差が出る」という知見が逆説的に確認できた。

3. **file-ops の重複パス**
   - `/Users/kawaz/src/github.com/kawaz/claude-nandakke/` (旧パス) と `/Users/kawaz/.local/share/repos/github.com/kawaz/claude-nandakke/main/` (新パス) の両方で同じファイルが Read されている。同一ファイルの 2 パスを Read する理由 (symlink 探索 or パス移行過渡期) は U 型なしでは確定できない。

4. **「No security findings」の信頼度**
   - T 型分析の詳細度は高く、各 recipe について明示的にシェルメタ文字エスケープを確認している。R 型の結論「No security findings」は T 型の分析の帰結として一貫しており、索引の確信列は「事実」とした (機械確認ではなくレビュー結論だが、内容が明示的)。
