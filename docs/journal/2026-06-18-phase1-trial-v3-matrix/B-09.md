## 3. 解釈に迷った点

### (1) `--width 9999` vs `--width 0`
`command_computed` に `--width 0` が記載されていた。`9999` を指定すると内部で `0`（無制限）に丸められる可能性がある。条件 B の「width 9999」が実際には「width 無制限」として機能しているなら、条件の意図通りかどうかは確認が必要。

### (2) セッションの日付とターン番号の対応
file-ops の timestamp は `2026-06-16` だが、timeline の `command_computed` にある `R58fd33b8..23 R2006cf20` はターン 1〜23 を指している。ターン 6（将来 issue 2 件起票）と ターン 13（snapshot JSON 実装）と ターン 17（auto-lock 実装）が大きな塊で、それぞれが「v0.7.0 / v0.7.1〜v0.7.2 / v0.8.0」のリリースに対応していると推定できるが、ターン対バージョンの正確な対応は timeline（R 行）の内容から読んだ（不確実さ残）。

### (3) 無関係 2 ファイルの扱い
ターン 22（R0516942d）で「`@` に他の issue 2 件 agent タスク外が居る」と言及がある。`docs/issue/2026-06-14-bug-suspend-resume-multidraw-claude-tui.md` と `docs/issue/2026-06-16-feature-session-labels.md` の 2 件。これらは file-ops では変更として出ていない（= このセッションでは Write/Edit されていない）が、jj の `@` には居た模様。session が触ったファイルかどうかの境界が曖昧で、索引に含めるべきか判断が難しかった。今回は「file-ops で検出されたものだけ索引化」とした。

### (4) regression の根本原因
ターン 23（Rc8de58a5）で `set_read_timeout` + `decode_from` の partial-byte-discard による frame decode 失敗の root cause 仮説が詳細に書かれているが、これがサブエージェントの調査結果を要約したものなのか、main が直接実装を読んで導いた結論なのかが R 行だけでは判別しにくい。

## 4. 自分の作業手順

1. 背景ドキュメント（DR-0001、ROADMAP、decisions-log）を Read ツールで読んだ。
2. `claude session timeline ...` を Bash で試したところ `error: unknown option '-t'` が返った。`claude` CLI には `session` サブコマンドが存在しないため、`claude-session-analysis` スキルのバイナリを直接呼ぶ必要があることを確認した。
3. スキルの SKILL.md（`/Users/kawaz/.claude-personal/plugins/cache/claude-session-analysis/claude-session-analysis/0.14.0/skills/claude-session-analysis/SKILL.md`）をスキル起動で確認し、バイナリパスを特定した。
4. `$SKILL_DIR/bin/claude-session-analysis timeline -t R --width 9999 269559b3-...` を実行。出力が 43.7KB で `<persisted-output>` に保存された。
5. persisted-output のパスを確認して `cat` したが再度 persisted-output になった（ループ）。Read ツールでファイルを直接読み、全 134 行（R 行 134 件）を取得した。
6. 同時に `$SKILL_DIR/bin/claude-session-analysis file-ops -d 1 269559b3-...` を並列実行し、変更ファイルの JSONL を取得した。
7. file-ops 出力（各行: timestamp / turn / tool / path）と timeline（R 行の内容）を照合して索引テーブルを構成した。
8. 出力先ディレクトリ（`docs/journal/2026-06-18-phase1-trial-v3-matrix/`）を作成し、Write ツールで本ファイルを書いた。
