## この条件で見えなかったこと

**T (Assistant 思考) + R (Result / ツール結果) を `--md=source` で読む条件。R 型が加わることで T のみの条件より情報量が増えるが、以下は取りこぼした。**

1. **ユーザの実際の発言原文** (U 型なし)
   - T 型は Claude の内部解釈であり、ユーザが何を・どの言葉で要求したかは直接見えない。
   - 例: turn 7 でユーザが「完了済み issue を削除せよ」と指示した文脈、turn 16 で bgIsolation の設定を要求した背景など、T 型の解釈文から推測するしかない。
   - turn 8 の冒頭「ユーザが指摘した通り、確認止まりで手を動かしていない」という T 型記述も、ユーザがどう指摘したかは不明。

2. **W 型 (assistant の実際の応答テキスト)**
   - T 型は Claude の思考プロセス、R 型はツールの出力であり、ユーザに返した文面 (W 型) は含まれない。
   - ただし本セッションの R 型エントリには「result: ...」で明示されたサマリが多く含まれており、W 型の代替情報として機能した部分がある。

3. **B 型 (Bash コマンドの実行コマンド文字列)**
   - どの Bash コマンドを叩いたか (例: `jj squash --into a1d941d4`、`just push-without-bump`、`sed` コマンド等) の詳細は T 型の文脈から推測するしかない。
   - 特に marketplace.json の修正を「Bash 経由で sed した」という事実は T 型 (T8ab4e98c) で把握できたが、実際のコマンド引数は見えない。

4. **F 型 (ファイル操作詳細)** を file-ops で補完
   - `file-ops -d 1` により Edit/Read ファイルの一覧は把握できた (marketplace.json, settings.json 等)。
   - ただし file-ops は tool 呼び出しを列挙するのみで、変更内容の before/after は T 型の文脈から読むしかない。

5. **EnterWorktree ガードの内部動作**
