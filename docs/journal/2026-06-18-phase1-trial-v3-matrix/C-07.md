## 解釈に迷った点

- **`-t TR` 条件での T 行の密度**: T 行は assistant の思考ログであり「ユーザの意図を読んで次に何をするか」の推論が詳細に出る。`-t R` 条件 (B-07) では「なぜここで突然 process substitution の話が出るのか」が謎だったが、本条件では T8a39c651「launchd 環境では process substitution が動作しない...」→ 直前の T で把握した上で R28ceab71 に流れていると追えた。T 行があることで因果が見やすい。
- **`any;+;` 正解の発見の経緯**: T の思考で「`iMessage;+;` が Sonoma 以降で縮小された可能性、chat.db の guid をそのまま使う方向」と推論してから Bash を叩いた流れが T431dc8a3 から読める。B-07 では「なぜその形式を試したのか」が不透明だったが、本条件では T で根拠推論が見えた。
- **SPA スタックの判断**: T0f65c926「クラスがアーカイブされた」「担任として招待された」という表示について「アーカイブされたクラスを継続監視している可能性がある」と推論しているが、この判断がその後覆されていない点が気になった。実際には /mc/ URL に切り替えることで解決しており、アーカイブ問題ではなかった模様。T の推論が必ずしも正確でないことの一例。
- **turn 3 の長さ**: file-ops を見ると turn 3 が最も file edit 数が多く、かつ Playwright を絡めた本文取得の試行錯誤も同 turn 内で行われている。T 行・R 行ともに多く、索引化作業的には「turn 3 での作業が多い」という情報は得られるが、どの T/R が「iMessage 実装」でどれが「本文取得調査」かを色分けする手がかりが timeline 上では明示されていない。
