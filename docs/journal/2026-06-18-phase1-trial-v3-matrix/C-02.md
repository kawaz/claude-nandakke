## -t TR で見えなかった内容

### 見えたもの vs 見えなかったもの

| 種別 | 見えた | 見えなかった |
|---|---|---|
| 推論の骨格 | T が長いので「なぜその判断か」が追える | ツール呼び出しの引数・返り値 |
| 決定の転換点 | T に「撤回」「修正」が明示 (B5 削除→昇格、--reveal→--no-masking) | 具体的なコマンド文字列 |
| サブエージェント結果 | R に要約が入る (codex/nitpick 指摘が数百字で再現) | subagent の内部思考 |
| ファイルの変更内容 | R に diff 相当の要約あり | 実際の before/after テキスト |
| 判断の流れ | turn 単位の文脈が追える | ツールが何を返したか (Read 結果等) |

### 具体的に欠けた情報

- `jj status` や `jj log` の実際の出力 (T に「この出力から判断」と書いてあるが内容は不明)
- op run 検証コマンドの実行結果テキスト (T に「予想外」とあるが生データ不明)
- subagent (nitpick/codex) への実際のプロンプト全文
- reviews-c/ 各ファイルの中身 (レビュー結果の詳細)
- commit hash と jj change id の対応 (T に一部出るが不完全)

### -t TR で足りる場面

セッションで「何が起きたか」「どの判断でどう変わったか」を追うには T だけで相当カバーできる。
特に R が結果要約を持つため、ツール呼び出し結果 (U) がなくても主要事実は取れる。

### -t TR が足りない場面

- ツールが返した生データをそのまま使いたい場合 (例: grep 結果の全件)
- subagent の生成内容を索引化したい場合
- 「T と実際のツール結果のズレ」を検出したい場合 (思い込み検出)

## 迷った判断

### TR だけで十分か、U も必要か

T が長くてリッチなセッションは TR でほぼ読める。
R が薄い (1行結果) セッションや、ツール返り値を直接索引化したい場合は U が必要。
本セッションは T が詳細で R も要約が充実していたので TR で追えた。

### 81KB の大ファイルをどう処理したか

`head -200` では 2KB のプレビューしか出ず、full output をファイルに保存してから
Read で 2 ページに分けて読んだ。93 行全部確認できた。
width 9999 は1行が長くなるため KB が大きくなりやすい。
索引用途では width を絞って行数を増やす方がトークン効率が良い可能性がある。

### file-ops の情報量

file-ops は「いつ・どのファイルを・どのツールで触ったか」のタイムスタンプ付きリストで、
turn との対応が取れる。TR timeline と file-ops を突き合わせると「T の判断 → 実際の変更」
の経路が追いやすい。file-ops 単独では「なぜ」が分からず、TR 単独では「何を」が薄い。
両者の組み合わせが索引として機能する。

### 気づいた挙動

- width 9999 は「折り返しなし」ではなく「9999 文字まで折り返し」の意。実際には
  --width 0 (no-limit) として処理されていた (command_computed に `--width 0` と表示)
- T が長いと1行あたりのトークンが多く、head での preview が 2KB でも数行しか見えない
- R (result) は Claude の最終出力要約なので、サブエージェント結果も R に集約されている
