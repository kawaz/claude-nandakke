## 解釈に迷った点

1. **セッション規模が大きく T 型の文章量も多い**
   - このセッションは 23 turn、2.4MB と本実験の題材の中でも最大級。T 型は各 turn で複数行の長い思考を含み、timeline 全体で 251 行の T/R エントリがあった。条件 A の T 型のみより情報量は大幅に増えたが、T 型の各発言が長くて重要な判断をスキャンするコストが高い。

2. **「相」の判断: implemented か recorded か**
   - ICANON chunk 化問題 (`feature-icanon-large-input-chunking.md`) と flaky test は issue 起票されているが、実装には着手していない。「spoken」ではなく「recorded」が正確だが、実装 issue として起票された段階で「相」がどこにあるかは議論の余地がある。今回は「spoken (issue 化)」として recorded に準じる扱いにした。

3. **DR-0022 の確信度**
   - T 型 `R398d01f0` で「実装完了、kawaz の MANUAL 編集と方向一致。push まで自律で進めます」とあり、その後 v0.8.0 push 成功 (`R71cf5c56`) と CI/Release 成功 (`Re3c1c473`) が R 型で確認できたため「事実」とした。ただし DR-0022 の具体的な設計内容 (protocol 番号、RAII guard の実装方法等) は T 型の思考記述から推定しており、実ファイルとの照合はしていない。

4. **codex stop-time review 由来の修正の「確信」**
   - turn 13 の `R6ab7556a` でヘルプ/README の古い記述修正が v0.7.2 でリリース済みと記述があり「事実」とした。ただし stop-time review は kawaz が session stop 時に有効化する機能で、「codex が指摘 → Claude が修正 → push」の経路は R 型の要約から読んだもの。

5. **claude-nandakke リポ起動の「implemented」は推定**
   - turn 22 の `Re82040a7` で「作成完了」の R 行があるが、turn 22〜23 は background 実行中の完了通知待ちという記述があり、実際に Claude プロセスが正常起動したかは最終確認できていない。`Tb07feea0` (turn 21) の T 型で「cmux-msg spawn は CMUX_WORKSPACE_ID 必須でこの方法では起動できない」という問題が発生し、代替として hyoui 経由に切り替えたという経緯がある。

6. **T 型 vs R 型の情報の質の違いが際立つセッション**
   - このセッションは複数の背景エージェント (bug 調査、実装、codex review、Workflow、MANUAL 更新) が並行して走っており、R 型にはエージェント完了後の統合要約が含まれる。T 型単独では「parallel agent を起動した」という記述止まりだが、R 型があることで「完了 + 成果の要約」が読め、索引の「実装されたか否か」の判断がかなり明確になった。
   - 例: `Rc46d8909` (turn 6) の R 行で「1017 pass / 0 fail」「partial-byte race 修正採用」「codex M1/M2/m1 全対応」が一覧で出ており、T 型だけでは得られない完了の証跡になっている。

## 自分の作業手順

### 手順

1. 背景ドキュメント 3 本を並行 Read (DR-0001, ROADMAP, decisions-log)
2. `file-ops -d 1 269559b3-fb71-4a89-bae6-95940a4aa65e` を 1 回叩いた (条件共通)
3. `timeline 269559b3-fb71-4a89-bae6-95940a4aa65e -t TR --width 9999` を 1 回叩いた
   - 出力が 143.6KB の persisted-output になったためファイルを Read で参照
   - ファイルが 252 行・53722 tokens で 25000 token 制限を超えるため offset 分割で 3 回に分けて読んだ (1〜99 行、100〜179 行、179〜251 行)
   - 追加で `-t` の種類を変えて再叩きはしていない
   - `--md` は付けていない
   - 範囲指定 (N..M) は今回は使わなかった (全ターンを読む必要があると判断)
4. 既存条件 C のサンプル (C-01.md) と A-01.md を Read してフォーマットを確認
5. timeline 出力 (T/R 行 計 251 行) を通読し、file-ops 出力と突き合わせながら索引テーブルを構築
6. 本ファイルに書き出し

### 条件遵守の自己確認

- `-t TR --width 9999` のみ: 守った
- `--md` なし: 守った
- 他 type (`U`, `W` 等) を追加で叩いていない: 守った
- `file-ops` は条件共通として 1 度だけ叩いた: 守った

### 特記事項 (セッション規模起因)

- 2.4MB / 251 行の timeline は persisted-output に保存され、直接バッファに返ってこなかった。ファイル Read で対応したが、1 回の Read では token 上限を超えたため offset を分割して 3 回読んだ。この分割自体は条件の `timeline` コマンドを複数回叩いたのではなく、出力ファイルの Read を分割したものであり、条件違反には該当しないと判断した。
