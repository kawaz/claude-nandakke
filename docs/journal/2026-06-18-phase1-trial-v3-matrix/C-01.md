## この条件で見えなかったこと

**T + R 型 (思考 + 結果/ツール実行結果) を読む制約で以下の情報が得られなかった。**

1. **ユーザの実際の発言内容 (U 型)**
   - R 型の先頭に「なるほど」「了解」「確認する」といった応答の断片は含まれるが、ユーザ原文は出ない。
   - T 型には「ユーザが〜と指摘している」という Claude の解釈が載るが、ユーザがどの言葉で問題を提示したかは不明。
   - 例: turn 2 の cwd/resume 指摘がユーザのどの発言から来たのかは T 型の要約からしか読めない。

2. **会話の流れのきっかけ (U 型 + W 型の欠落)**
   - turn 4 で突然「複数 range / --grep 複数指定」が議題になるが、ユーザがどのタイミングどの言葉でそれを要求したか、T/R 型だけでは追えない。
   - turn 5 の `-m N` も同様。「どこでユーザが要求した」の起点が見えない。

3. **R 型に含まれる情報の粒度差**
   - R 型は Claude の結果まとめテキストで、ツール実行の生ログ (テストの fail 詳細、Bash コマンドの stderr 等) は含まれない。
   - T 型に「テストが失敗している」と書かれているが、どのアサーションが落ちたかは R 型では確認できない。

4. **push 実施の有無**
   - turn 3・4・5 それぞれの末尾 R 行で「push (bump 含む) は指示待ち」という記述があるが、その後 push が実行されたかどうかは T/R 型には現れなかった (file-ops でも Read/Edit のみ、Bash の push コマンドは別途確認が必要)。

## 解釈に迷った点

1. **「--grep 複数 AND 確認済み」の確信度**
   - `R440520f2` で「`[20 hits / 2 kw]` 等」という実機出力断片が R 型に含まれていたため、これを「事実」とした。T 型のみの条件 A と異なり、R 型があることでツール出力の断片が索引の確信度判断に使えた。

2. **CI 全通過の確信度**
   - `Rca531166` に "CI 通過" と記述があり、これは Claude が確認した結果の要約なので「事実」と判断した。ただし R 型の記述は Claude の解釈を介しているため、完全な一次記録ではない点は注意が必要。

3. **「最終言及」に turn 内のどの R 行を使うか**
   - 同一 turn 内に複数の R 行がある (例: turn 3 は `R00e1dc7a`, `R7ee9ea11`, ..., `R1591c079`)。「最終言及」として turn 番号を採用したが、R 行の marker ID を使えばより細かく追えることが分かった。実用上は turn 番号で十分かもしれない。

4. **T 型と R 型の役割分担が明確**
   - T 型: 判断・解釈・設計思考 → 「なぜそうしたか」がわかる
   - R 型: 結果まとめ・実機確認断片 → 「何ができたか」の証跡になる
   - 両者を合わせると「判断の根拠」と「完了の証跡」の両方が揃うため、T 単独より索引の確信度が上がった。この条件が索引作成に向いている可能性がある。
