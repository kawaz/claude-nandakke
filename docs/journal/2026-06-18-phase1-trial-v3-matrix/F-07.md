## 見えなかったもの (-t TR --md=source の限界)

- **Tool 呼び出しの詳細が不可視**: `T` (思考) と `R` (応答) だけでは、実際に何のコマンドを実行したか、どの SQL を叩いたか、どのファイルをどう編集したかが分からない。例えばターン 3 で worktree を作成・編集していることは file-ops から分かるが、その具体的なファイル内容は TR だけでは追えない
- **実機テストの試行錯誤の細部**: `-1728` エラーが出た → 別形式を試す → `any;+;` が通った、という流れは思考ログ (T) で読めるが、「何種類の形式を試したか」「各試行の正確なコマンド」は R の要約のみで詳細が欠落している
- **Playwright のスナップショット/DOM 内容**: Classroom の詳細ページで本文がどの要素に入っているか、実際の HTML 構造が一切見えない。思考 (T) での推測の根拠となった観測値が不可視
- **chat.db の実際のクエリ結果**: どの行が返ってきて chat_id をどう特定したかの生の SQL 結果が不可視

## 迷ったところ

- **`-t TR` の T と R を区別しながら読む**: T が「Claude の思考プロセス (内部判断)」で R が「実際の応答 (出力)」。T の中に「私がこうしようと思う」が書かれていても、R で実際に何を出力したか確認しないと一致しているかが分からない。特に T で「過剰修正だった」と気づいた後に R で「修正を戻した」という流れは T→R ペアを対応させないと追えない
- **--width 9999 の効果**: width 制限なしのため 1 行が非常に長くなり、読みとしては楽だが出力ファイルが大きくなった (69.2KB)。width 制限があれば折り返しが入ってより「見やすい」が、折り返しで意味の境界が壊れることもあるため、索引作成目的では 9999 の方が意図を誤読しにくい
- **--md=source の意味**: `render` との違いを意識しながら読む必要があった。`source` ではコードブロックや見出しがマークダウンのまま出力されるため、読み取りは「生テキスト」として扱える。コードブロック内のコマンド等をそのまま抽出できた
