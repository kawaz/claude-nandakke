## この条件で見えなかったこと

**`-t R --md=source` (ツール結果のみ) を読んだ場合の特徴と制約。**

1. **Claude の判断過程が一切見えない**
   R 型はツール実行の結果だけを返す。なぜそのコマンドを実行したか、どういう判断でコードを変更したかは R 型には現れない。索引テーブルを作るにあたって「この変更の意図」は R ブロックの内容から逆算するしかなかった。たとえば `input.workspace.repo` への切り替えが「worktree パスで parseRepo が壊れる」問題への対処だったことは、R ブロックに dump の `workspace.repo` フィールドが現れたことから辛うじて読み取れたが、確証は薄い。

2. **ユーザ発言の内容が取れない**
   U 型を読んでいないため、何を指示されて次の作業に移ったかが分からない。「よろ」「とりこみ」のような短い指示が何ターンにあったかは R 型の文脈切れから推定するしかなく、相の判定が「推定」止まりになるケースが増える。今回はターン番号と時刻が手がかりになった。

3. **dump ファイルの生 JSON は R 型から読み取りやすい (これは利点)**
   R 型は Bash ツールの stdout / Read ツールの内容をそのまま含むので、dump 解析の結果 (statusbar JSON フィールド一覧) は R ブロックに詳細に出てきた。条件 D (T 型) では「T 型の記述から推測するしかない」状況になっていたのと対照的で、実際の調査内容を索引化する用途では R 型のほうが具体的なエビデンスを取りやすい。

4. **OSC 8 問題の議論の流れが取れない**
   turn 8-9 で OSC 8 リンクと Ghostty のマウストラッキングについて長い説明が交わされたが、それは R 型 (ツール実行結果) ではなく Claude の応答文章なので R 型には現れなかった。問題が「解決した」か「脇に置いた」かは、dump 解析へ移行する Bash ツール呼び出しの流れから間接的に読み取るしかなかった。

5. **bgIsolation guard のやり取り詳細**
   turn 7 で「background session の guard が shared checkout への edit を reject した」というエラーが発生し、worktree や settings の切り替え議論が複数ターン続いた。この議論は Claude の応答文章であり R 型には現れないため、worktree 隔離に至った経緯は R ブロックの Edit/Read の連続から逆算するしかなかった。

2. **`input.worktree` フィールドの相**
   turn 22 の R ブロックに dump の JSON 内容が出てきたとき、`input.worktree` フィールドの存在が「発見」された瞬間だと判断した。しかし R 型だけでは「Claude が事前に知っていたのか、dump を見て初めて気づいたのか」が分からない。T 型 (条件 D) では「完璧な手がかり」と Claude 自身が書いていることが分かるが、R 型ではその驚きのニュアンスが取れない。

3. **turn 34 の dump 解析が「観測」か「設計」か**
   turn 34 で cost フィールド一覧が R ブロックに現れた。これはすでに dump ファイルを Read した結果なのか、Claude が提案として列挙したテーブルなのか、R 型だけでは区別しにくかった。実際は dump の input.json を Read した結果だった (= 事実として索引化可) が、確証を得るために R ブロックの前後文脈 (Bash ツールの Read 結果として現れているか) を確認する必要があった。

4. **turn 39 (セッション末端) の状態**
   最後の R ブロックは dump 仕込みの CI 確認 (tsc / test pass) だった。セッションはここで終わっておらず観測継続中のはずだが、timeline はここで終了している。「観測の途中でセッションが止まった」のか「その後の観測データがまだ jsonl に書かれていないだけ」なのかは R 型からは判断できない。

### `-t R --md=source` の効果に関する観察

R 型 (ツール実行結果) はコードの変更内容・コマンドの出力・ファイルの読み取り結果を直接含むため、「何が行われたか」の事実確認においては T 型より具体的なエビデンスが取れた。特に dump ファイルの JSON 内容が R ブロックにそのまま現れる点は索引化しやすかった。

一方、「なぜそうしたか」「Claude がどう判断したか」「ユーザが何を指示したか」は R 型だけでは取れない。会話の流れや分岐 (例: worktree を使うか main を直接触るか) の判断経緯は欠落する。

`--md=source` については、R 型のツール出力には Markdown コードフェンスやテーブルが多く含まれており、`--md=render` (デフォルト) との差分は今回の用途では有意だった。レンダリングされると表の境界や構造が人間向けに整形されてしまい、AI が後処理で解析する場合には source の方が扱いやすい。
