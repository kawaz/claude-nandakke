## この条件で見えなかったこと

**`-t T --md=source` を組み合わせた場合の特徴と制約。**

1. **`--md=source` の効果**
   - 条件 A (`--md` なし) と比較して、T 型の出力内容自体は同じ。`--md=source` は T 型の Markdown をレンダリングせずそのまま返すフラグで、今回の T 型 (Claude の思考) には Markdown 記法がほとんどなく、効果の差分は体感できなかった。
   - 差分が出るとすれば W 型 (会話ブロック) や R 型 (ツール結果) に Markdown コードブロックが含まれる場合だが、今回は `-t T` のみで U/R/W を読んでいないため判断できない。

2. **ユーザの実際の発言 / 指示の内容**
   - A-01.md と同様に T 型だけでは「ユーザが何を言ったか」が Claude の解釈越しにしか取れない。特に「よろ」「とりこみ」などの短い指示がどのターンにあったか、T 型からは推定に留まる。

3. **dump ファイルの実際の内容**
   - turn 7-22 にかけて `~/.cache/claude-statusline/debug/` に dump が落ちていたはずだが、実際の JSON 内容は T 型だけでは見えない。`input.worktree` フィールドの内容 (`name`, `path`, `branch`, `original_cwd`) が確認できたのは T 型の記述から推測できるが、生データは B 型や R 型を見ないと確認不能。

4. **OSC 8 / Ghostty 問題の解決有無**
   - turn 8-9 で OSC 8 リンクと Ghostty のマウストラッキング問題が議論されたが、「解決した」か「未解決のまま別問題に移行した」かが T 型だけでは判別しにくい。「ダンプの仕込みを残すか戻すかを確認」という記述から、OSC 8 問題は脇に置いて dump 調査に移行したと推定できるが確証は薄い。

5. **`just push` の実行ログ詳細**

### `--md=source` の効果に関する観察

T 型 (Claude の思考ブロック) では Markdown 記法がほとんど使われていないため、`--md=source` と `--md=none` (または `--md` なし) の出力差分は今回のセッションでは事実上なかった。`--md=source` が有意差を出す場面は W 型 (会話) や R 型 (ツール出力) で Markdown コードフェンスや箇条書きが多い場合と考えられる。条件比較において `-t T` と `--md=source` の組み合わせは「効果が重ならない」可能性があり、`--md=source` の効果を見るなら `-t W` や `-t R` と組み合わせる方が適切かもしれない。
