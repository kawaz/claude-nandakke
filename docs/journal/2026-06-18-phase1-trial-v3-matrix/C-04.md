## 解釈に迷った点

1. **T 型と R 型の情報量差**
   - 条件 A (T のみ) と比べて R 型が加わったことで情報量が大幅に増えた。特に R 型には実走結果の引用 (OSC 8 バイト列、CI 結果のフォーマット出力、dump の JSON 抜粋) が直接掲載されており、「事実」判定に使いやすかった。
   - 一方 T 型が長く詳細な場合、T と R で同じ結論が繰り返される冗長さがあった。

2. **「相」の判定: implemented か recorded か**
   - 今セッションの多くの topic は R 型に「push 完了」「CI success」「commit 取り込み完了」という明示的な記述があり、implemented と判定しやすかった。
   - `input.worktree` / `input.workspace.repo` の発見はコード修正を伴わない情報収集なので「recorded」とした。

3. **cost Δ$ 問題が解決済みか未解決か**
   - セッション末尾 (turn 38-39) は「観測モード突入」で終わっている。T 型に「ターン境界識別が必要」という分析はあるが、実装した形跡 (Edit や push) は timeline 末尾 (Tc99ba21d, turn 39) には見えなかった。「recorded」(問題認識・方針策定済み) だが「implemented」ではないと判定した。

4. **`release-flow-awareness.md` 編集の cross-repo 性**
   - このセッションは claude-statusline リポ作業中に、claude-rules-personal リポのルールファイルを直接 Edit した。file-ops で `/Users/kawaz/.local/share/repos/github.com/kawaz/claude-rules-personal/main/for-me/rules/release-flow-awareness.md` への Edit を確認 (turn 27)。
   - 条件 C では R 型にルール改定の提案→承認→実装の流れが読め、turn 28-32 の T 型で cross-repo commit/push の具体手順の思考が見えた。

5. **dump 仕込みの「残存」扱い**
   - turn 39 末尾時点で dump コードが main の `src/statusbar.ts` に残ったまま観測モードに入っている。これを「implemented」と扱うか「wip」と扱うかが迷い点。本索引では「観測中 (セッション終了時点での状態)」と注記した。
