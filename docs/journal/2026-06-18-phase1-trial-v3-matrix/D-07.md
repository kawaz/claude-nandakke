## この条件で見えなかったこと

条件 A (-t T / md なし) との差分に注目して記述する。

- **条件 A と読み取れる情報量は変わらなかった**: `--md=source` を付けても、T 行 (思考サマリ) の内容は条件 A と同一。T エントリはもともと Markdown ソースとして書かれており、`--md=source` による差分が生じなかった。コードブロック・箇条書きが Markdown のままレンダリングされずに出るという変化はあるが、情報量の変化はゼロ。
- **会話の生テキストは依然ゼロ**: ユーザー発言・Claude 回答の文面は引き続き見えない。T 行は各ターンの思考サマリのみ。

- **Playwright の snapshot / DOM 構造**: T2a43216d, T66e982b1, Tbc702def 等で DOM 探索の経緯が語られるが、実際の HTML 断片は見えない。これも条件 A と同じ。
- **実際のコード diff**: file-ops で Edit の実績は分かるが、変更内容はブラックボックスのまま。

## 解釈に迷った点

- **`--md=source` の効果対象**: T 行のテキストは元から Markdown 記法なので、`--md=source` は「レンダリングせずソースのまま出す」という意味になる。条件 B (--md=render) と比べると、コードブロックや見出しが Markdown 記号のまま出るかレンダリングされるかの差が生じるはず。T 型のみの場合、その差が索引作成の作業感に影響するかは、本実験では B との対比がないと判断しきれない。
- **条件 A との本質的な違いの薄さ**: `-t T` 限定では T 行の内容が Markdown ソースとして出てくるため、`--md=source` / `--md=render` / md なしの違いが最小になる。R (回答) や U (ユーザー) を含む型では Markdown レンダリングの差が効いてくると予想される。
- **turn 3 の密度**: 条件 A と同様、turn 3 に T エントリが 12 個以上集中しており、1 ターンの定義が通常と異なる可能性がある。`--md=source` オプションではこの密度の理由を解明する手がかりは増えない。
