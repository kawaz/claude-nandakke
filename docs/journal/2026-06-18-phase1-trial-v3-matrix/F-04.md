## この条件で見えなかったこと

**-t TR (思考 + 応答) / md=source** の縛りで生じた情報の欠落:

- **ユーザ発言の具体内容が見えない**: T 列と R 列だけでは AI の思考・返答しか見えず、ユーザが実際に打った文字列は見えない。「よろ」「パッチで良い」「とりこみ」等の短い指示語は R 列から間接的に読み取れる (例: R428cb797 「取り込みます」からユーザが「とりこみ」と指示したと推定) が、ユーザ側の原文は確認できない。
- **ツール実行結果の生データが見えない**: Bash の標準出力・Edit の差分・Read した内容が全く見えない。R 列で「dump の出力確認」「CI 全通過」等を語っているが、それに基づく根拠データ (実際のコマンド出力・ファイル内容) は追えない。索引の「確信=事実」は R 列での言及を根拠にしているが、生データの直接確認はできていない。
- **T 列で展開された検討の枝葉**: 今回の条件では T 列 (思考) も含まれているため、設計検討の内部モノローグは見えた。ただし T 列が R 列より詳細な検討内容を含む場合に、どちらが「確定情報」かの判断が曖昧になった (例: Tdc3f8cf8 の shim 分析は T 列での推論であり、R 列での確認とは重みが異なる)。
- **ツール呼び出しの引数・返り値**: A 条件 (T のみ) と同様、ツール実行の具体内容は見えない。file-ops でファイルパスは分かるが、変更内容は不明。
- **コスト・ context_window の時系列変化**: R 列の末尾まとめ (R91547ae9) で「$6.85」等が出てくるが、それ以前のターン単位の推移は読めない。
- **環境変数・設定値の実値**: CLAUDE_CONFIG_DIR, GHOSTTY_VERSION 等の実値は T/R 列に出てきた場合のみ読み取れる。

## 解釈に迷った点

- **「main 直編集」の許可と bgIsolation ガードの関係**: R 列では「ユーザが worktree 不要と指示 (Re38e443e)」→「guard に弾かれた (Re304d002)」→「settings.local.json で無効化 (Re3b1c3c0)」→「鶏卵問題で Bash 直書き (Ra5d098c6)」という経緯が読み取れた。ただし最終的に guard が無効化されたかどうかは T 列の推測と R 列の報告が一致せず (「settings.json 側で読まれてる (Rbe4eacf1)」「session 中 reload されない (T3708039a)」)、実際に guard が外れたのかどうかが R 列だけでは曖昧。file-ops で turn 34 に main/src/statusbar.ts への Edit が記録されているので結果的に書けたと判断した。
- **debug dump の最終残存状態**: セッション turn 33 で「worktree・dump を撤収 (Rfd358417)」とあるが、turn 39 末尾で「再仕込み+ci 通過 (R8e0ea8f3)」が確認できる。R 列の "完了" 系の報告と後続の "再仕込み" が矛盾なく読み取れるのが TR 条件の強み (A 条件の T のみでは両者の文脈がつながりにくかった)。
- **v0.1.2 と v0.1.3 の commit 内容分割**: R43d38bb9 で「feat(statusbar): worktree badge + green git branch」、Ra5b1eb15 で「v0.1.3 push」と明示されており、TR 条件では R 列から commit メッセージを直接読み取れた。A 条件 (T のみ) より確認しやすかった。
- **R 列の "了解" 系の前後関係**: R 列には「進めます」「了解」「実装します」等の短い確認応答が多く、それ単体では何を指しているか T 列との対応なしに文脈を取りにくい (例: R3a764c24「進めます。実装: 1. input.workspace.repo 優先... 2. input.worktree?.name あれば 🌿バッジ」は T 列 Ta12b18f9 の承認後の実装宣言)。TR 条件では T+R 両方から前後関係を追えるため A/B 条件より有利だったが、R 列だけ読んでも文脈が取りにくい箇所が複数あった。

5. T 列とR 列を組み合わせて設計検討→決定→実装の流れを追跡し、file-ops で実装の有無を補完した。確信列は「R 列で具体的な実行結果・生データを引用した場合は事実、推測・計画段階のみは推定」で判定した。
