## 解釈に迷った点

1. **`--md=source` が索引テーブル作成に与えた影響**
   - 通常の `-t T --width 9999` (条件 A 相当) と比べると、`--md=source` によって T 型の内容が markdown の raw ソースとして取得できた。T 型は元々 markdown 記述が多い (コードブロック・リスト・テーブル等) ため、render 済みより source の方が構造が読みやすかった。
   - ただし、`--md=source` の効果がどこまで影響するか (T 型以外の型ではどうか) を今回の条件では確認していないため、条件 A との差異は限定的かもしれない。

2. **「implemented」の境界: commit 済み vs push 済み**
   - turn 21 末尾で push → turn 27 で divergent 発覚 → turn 28 で調査 → turn 29 でリベース後再 push という複雑な経緯があった。最終的に push されたと T 型には記述があるが、「implemented」は「セッション内で実装完了かつ main に反映済み」という意味で使っている。turn 31 まで確認が続いているため push 成功の確度は高いが、確信は「事実」ではなく「推定」とした。

3. **C6-C9 の案件修正の最終コミット対応**
   - C6 は turn 21 で commit (`uuqvrozz 05326938`)、C7/C8/C9 も turn 21 内で順次 commit された。しかし turn 27 でさらに修正 commit (`vstqvyvl`) が積まれ、それが turn 29 のリベースで main に統合された。索引テーブルの「最終言及」は最終確定となる turn 27 を指すべきか、最初の commit ターンを指すべきか迷った。今回は両方記載した。

4. **authsock-warden への起票が「ネットゼロ」になった件**
   - T 型の記述から「issue ファイルを作成 → kawaz 指示で削除 → ネットゼロ」と読み取れるが、削除の理由が「Note が古くなったため上流還元不要になった」という解釈だけでは不十分かもしれない (turn 21 の T3d631ca1 に「kawaz の修正方針が変わり Note 全削除」とある)。上流還元の方針自体が変わったのか、この case では不要だったのかが T 型だけでは断言できない。「相: spoken / 事実」で記録したが「実質的に何も起きなかった」扱いが適切か迷った。

5. **87KB 超の timeline が persisted-output に保存された件**
   - 今回の timeline は 87.4KB に達し、tool-result として自動保存された。条件 D の `--md=source` が出力量に影響した可能性がある (通常の `--md=none` より markdown の raw ソース部分が多い?)。他条件との比較には別途検証が必要。
