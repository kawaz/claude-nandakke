## この条件で見えなかったこと

- **ユーザ発言の具体的なテキスト**: `-t R` は応答 (Response) ノードのみ取得する。turn 7 で「手を動かしていない、確認止まりでは?」のような指摘があり方針が転換したことは応答から読み取れるが (Reca6d5a6 「おっしゃる通り、確認止まりで手を動かしてませんでした」)、ユーザが何を言ったかの原文は含まれない。
- **思考プロセス**: `-t T` (思考) を含まないため、各判断の途中の推論ステップが見えない。例えば「noise-classification の PR③④不要」と判断した根拠の詳細、check-bundle を ci から外す設計判断の経緯は応答の結果だけが見える。
- **ツール入出力の詳細**: Bash コマンドの実行結果や Read の内容は `-t R` には含まれない。CI failure の原因が「GH_TOKEN 未設定」であることは応答テキストから読み取れるが、実際のエラーログ原文は不可視。
- **中間の tool 呼び出しシーケンス**: file-ops では turn 5〜7 で Read が 3 件 + Write が 2 件 + Edit が複数だったが、timeline では応答の要約だけが見える。並列実行の様子や retry の有無は不明。

## 解釈に迷った点

- **`--md=source` の効果の評価**: 出力は Markdown ソース記法 (ヘッダ `#`, テーブル `|`, コード `` ``` ``) がそのまま残り、render された HTML ではない。nandakke の索引作成用途 (= AI が構造化データを読む) では source のほうが扱いやすいが、人間の素読みには若干ノイズが増える。どちらが「目的に合っているか」は用途依存で、条件単体での評価は難しい。
- **turn 8 の情報密度の高さ**: R7a2ecec9 (turn 8) に「やったことサマリ」として 4 点が列挙されており、索引作成には便利。一方で turn 8 が 1 応答に多くの action を詰め込んでいるため、本当に全 action が 1 ユーザ発言に対するものなのか、複数の発話が圧縮されているのかが `-t R` だけでは判断しにくい。
- **rate limit エラー (Ra291e910)**: turn 16 のレスポンスに "API Error: Server is temporarily limiting requests" が含まれる。これが indexing の観点でノイズ (= 内容なし) か、それとも「このセッション中にレート制限が発生した」という事実情報として記録すべきかの位置づけが曖昧。
- **Reca6d5a6 の解釈**: 「おっしゃる通り、確認止まりで手を動かしてませんでした」とあるが、これは何かに対する謝罪的な応答。前のユーザ発言が見えないため、どの点で「確認止まり」だったのかの文脈が不完全。ただし後続応答 (R0750bcec) で「削除不可」の結論が覆されていないため、方針の実質的な変更ではなく進め方の指摘と推定できる。
