## 見えなかった

1. **T エントリで推論の「足場」構造が可視化される**: `-t R` (E-02) では最終応答しか見えないが、`-t TR` は T エントリで「何をどう判断しようとしていたか」の中間過程が読める。例えば turn 6 では「4 並列横断評価」の R 応答の前に、T エントリで「codex 2 件 + nitpick 2 件の severity 分類を考えている途中」が残っており、横断表の根拠となった思考経路を確認できる。R だけ読むと「表がある」、T まで読むと「その表をどう組み立てたか」が見える。

2. **rebase 誤手順の T エントリが特に有益**: turn 27 で `jj rebase -s lpxzktkm -d main` を試みて conflict が発生する経緯は、R エントリだけでは「失敗した → 報告」となる。T エントリ (`T502fd172`) を見ると「lpxzktkm が 6 件の途中だと気づかず commit 群の先頭でないものを指定した」という認識の遅れが見え、なぜ `uuqvrozz` から切り出すべきだったかの理解経路が追える。

3. **`op run --reveal` の推測継続**: turn 16 実機検証前に複数の T エントリで「`--reveal` フラグを案文に含めたまま書いた」形跡がある。R エントリには案しか出てこないが T エントリには「kawaz の言っていた `--reveal` で...」という文脈が入っており、事実確認より先に案を進めた手癖が T エントリに残っていた。

## 迷った

1. **T/R 二重化の読み負荷とサイズ**: `-t TR` は `-t R` に比べてサイズが約 1.7〜2 倍になる。turn 6 (4 並列横断評価) のように T と R が両方長いターンは特にコストを押し上げる。一方で turn 1 のような「ほぼ行動のみのターン」では T がわずか 2〜3 行で R が中核という比率になり、T の追加コストはほぼゼロ。「T が短い turn は T でも損しない、T が長い turn こそ T を読む価値がある」という逆転関係がある。これを事前に判断できる指標 (= T/R 比率の per-turn 表示) があれば取捨選択が楽になる。

2. **T エントリの「保留判断」が R に出てこないケース**: turn 28 の T エントリ (`T022cbd26`) では「ファイル単位重複なし → `jj rebase -s uuqvrozz -d main` で行けるはず」と内部で結論付けているが、R エントリ (`Rbb0bbb0c` / `R6d120e21`) では kawaz への確認として結論を留保している。T が「決まっている」のに R が「仰ぎます」になっている構造は、T だけ読むと「なぜ確認するのか」が疑問になるが R まで読むと「`discussion-style` の方針未決で走るなルールに従った」と理解できる。T と R を両方読まないとこのような「意図的な留保」が見えない。

3. **`-t TR` vs `-t TRU` の境界**: `U` (tool call/use 結果) を除いているため、turn 21 の多数ファイル編集 (20+ ファイル操作) は T/R では「何を変えたかの要約」しか見えず、個別の diff は `file-ops` に頼ることになる。今回は `file-ops -d 1` を条件共通で別途叩いているのでカバーできているが、T/R だけで変更内容の詳細を確認しようとすると `U` の欠落が問題になる。

## 手順

```bash
# timeline 取得 (条件 F: -t TR / width 9999 / md=source)
claude-session-analysis timeline f541f246-6128-4e2b-81b7-48a91baaefc9 -t TR --width 9999 --md=source

# file-ops (条件共通 -d 1)
claude-session-analysis file-ops -d 1 f541f246-6128-4e2b-81b7-48a91baaefc9
```

**`-t TR` の実際の出力形式**:  
`command_computed:` を見ると `-t TR --width 0` に正規化された (= 9999 は「無制限」扱い)。  
出力は各 turn の T エントリ (内部推論テキスト) と R エントリ (最終応答テキスト) を  
タイムスタンプ + ターン番号付きで交互に並べた形式。同ターンの T は R より前に出る。  
`--md=source` でコードブロック・表がそのままマークダウン文字列として出力される。

**読み取りコスト感**:  
E-02 (`-t R`) は全 31 turn で約 65 KB・1450 行。`-t TR` はその 1.7〜2 倍が目安。  
T エントリは 1〜3 行のものから 200 行超えのものまで幅がある。  
turn 6 (4 並列横断評価) は T も R も長いため単独でかなりのサイズを占める。  
`--width 9999` の折り返し解除は T エントリにも効くため、長い推論が 1 行で収まり  
grep で引っ張りやすい反面、1 エントリあたりの行数が T だと R より読みやすい粒度に収まることが多い。  
T エントリは「意図と判断根拠」、R エントリは「実際の出力」と役割が明確に分かれており、  
timeline を `grep -A 3 '^2026-06-17T11:4'` のように時間帯で切り出す際にも T/R の対応が取りやすい。
