# DR-0001: プロジェクト知識の索引化サイドカーのアーキテクチャ

- status: accepted (初版 / 実装前)
- 起票: 2026-06-17
- 対象: 新規プラグイン (名称未確定 — §7)

## このツールの本質 (一行定義)

**AI に、人間が持っている「ぼんやりした正確な全体インデックス」を持たせる。**

人間 (kawaz) は、各プロジェクトで何をどの辺で話し・決め・作ったかを、
正確ではないが「ぼんやり全体」として記憶している (= 索引がある)。だから
「探せば出てくる」と確信して探せる。AI にはこの索引がない。AI が全体を
把握するには docs とコードを全部読むしかなく、重いから端折る → 読まない →
想像で書く (cache-warden の事故) が起きる。**全部読む (重い) か 読まない
(雑) かの二択**しかないのが問題で、人間が持つ中間 ——「ぼんやり全体を把握、
必要なら正確に取りに行く」—— が AI には欠けている。本ツールはその中間を
AI に与える。

## 背景 / 解くべき問題

記録 (外部記憶) は既に十分ある — docs / コード / DR / findings / journal。
足りないのは記録ではなく、その上に乗る**索引**。索引がないことで:

1. **AI が全体を把握できず端折る** — カバーするには全部読むしかなく重い。
   結果、読まずに想像で進める (cache-warden の「aw でも同じはず」型の事故)。
1. **アイデアの相が追跡できない** — ある決定が「話しただけ (spoken)」
   「DR/ファイルに落ちた (recorded)」「実装まで行った (implemented)」の
   どれかを把握しないまま進み、数日後に「これこうなってたはず」と見ると
   DR にも実装にもない齟齬。復旧のたびに過去セッションを keyword で漁る。
1. **セッションの現在地を忘れる** — 放置した並列セッションに戻ると何を
   していたか失われ、メインに「今何中?」と聞く。メインは序盤が揮発して
   いるため不正確かつコンテキストの無駄。

根本原因は、全体把握をコンテキスト (= 揮発する記憶) に依存していること。
対策はルール (規範) ではなく、索引を外部化する仕組み (構造) で行う。

## 決定

セッションのコンテキスト外で動くサイドカーを新規プラグインとして作り、
**プロジェクト知識の索引**を維持する。索引は内容のコピーではなく
**ポインタ** (どこに何があるか)。粗くてよい — 人間の記憶が「正確じゃない」
のと同じで、当たりが付けば十分。完全網羅も最新同期も求めない (重くなる)。

役割は3つ、いずれも「AI 用の外部索引」という単一の核の異なる面:

- **記録 (write)**: 会話・指示・アイデア・作業・ツール利用の流れを継続
  把握し、決定とその相、分岐を中央の索引に永続化する。
- **自己点検 (verify)**: 索引の未確認 (相が推定どまり) を、副作用ゼロの
  確認 (リポの ls/grep) で事実化する。重い確認 (他セッションへの問い
  合わせ) は承認制。
- **応答 (serve)**: AI がセッション開始時に索引だけ読んで全体を掴み、
  詳細は索引から辿って必要分だけ正確に読む。現在地参照にも答える。

### コアは索引であって可視化ではない

「有機的グラフ」「時系列軸/アイデア軸のビュー」「ブラウザ可視化」は
**非コア**。コアは AI が当たりを付けるための軽い索引。可視化は「あれば
嬉しい」止まりで、最初に作るものではない (Understand-Anything 的な
リッチ可視化に引っ張られないこと)。

### スコープ境界

- **対象**: 各プロジェクトのプロジェクト知識 (DR / findings / journal /
  docs) と、それを生むセッションの会話・指示・アイデア・作業。
- **対象外**: ルール群 (claude-rules-personal の for-all/for-me)。
  ルール = 規範 (どう振る舞うべきか) は別ドメインであり、本ツールの
  プロジェクト知識 = 記録 (何が起きたか/分かったか) とは混ぜない。
  「docs」と書いても各リポのプロジェクト知識を指し、ルール群は含まない。
  この境界を明記するのは、後で「docs 全体」を拡大解釈してルール群まで
  飲み込む実装ドリフトを防ぐため (= 本ツール自身が防ぎたい事象)。
- 横断するとしても「複数プロジェクトのプロジェクト知識」まで。

### 第一段での検証ポイント

索引が「全部読む」の代替になるか = **手動で作った索引を AI に渡したら、
全部読まずに当たりを付けて正確に辿れるか**。これが効けばツールの価値が
実証される。効かなければ可視化も自動化も土台がない。

### アーキテクチャ

```
メインセッション (長時間・序盤揮発)
  │ 差分供給 (CSA messageid 以降指定 — §6 未検証)
  ▼
追跡セッション (1 コンテキスト継続・流れを保持・相と分岐を判断)
  │ 確定した相を書き出し、自身の文脈からは手放す (要約落とし — §6)
  ▼
中央記録層 (永続・横断検索可能)
  ▲
  │ 振り返り (pull・任意のタイミング)
メインセッション ← スキル or ブラウザ表示
```

- **追跡セッションが継続文脈を持つ**のが要。差分 (CSA) 単体では断片で
  判断できず、継続文脈に流し込んで初めて「この分岐はさきほどのアレの
  続き」と相・経路を判断できる。メインが序盤を忘れるのに対し、追跡は
  記録に徹して文脈を保つ。ただし追跡も無限ではなく、溢れる前に確定相を
  記録層へ落とす (§6)。
- **記録層は中央配置**。アイデアは複数プロジェクトをまたぐため、
  プロジェクト内ファイルでは横断検索 (keyword 漁り) が叶わない。
  永続化形式 (中央 DB / ディレクトリ+ファイル) は §6 未決。
  cmux-msg の “file-based truth vs SQLite for inotify-driven delivery”
  と同型の判断であり、そこでの結論を流用する。

### 索引スキーマ (6 列)

手動試行 (2026-06-17、チャットセッションで第一段を実演) で確定:

|列    |内容                                         |
|-----|-------------------------------------------|
|topic|決定・議論の主題                                   |
|相    |spoken / recorded / implemented            |
|確信   |事実 (機械確認済) / 推定 (追跡セッションの判断)               |
|最終言及 |最後に触れた時点 (実装では messageid/timestamp)        |
|参照   |セッション uri 等のポインタ (内容は重複させない)               |
|next |次に進めるべき相、または確認手段 (例: `[確認] ls docs/issue/`)|

`確信` 列は必須。追跡セッションの判断 (推定) と機械確認済 (事実) を
混同して記録すると、後で振り返る人間が嘘の地図を見る。相を倒すのは
事実が取れた時だけ。

### 現在地応答とメッセージ差し込み

- **第一候補: ブラウザ常時表示**。現在地は「質問」ではなく「参照」で
  あり、会話ターンを消費する必要がない。記録層を表示する静的ビューを
  用意し、セッションごとの URL をステータスバーに置く (statusline /
  Tampermonkey の既存資産)。これなら会話への差し込み経路自体が不要。
- **会話への差し込みが必要な場合: モニタ経由のソケット**。cmux-msg が
  既に行っている機構を再利用する。新規の通信インフラは作らない。
- UserPromptSubmit フックで現在地質問キーワードを検知し、`exit 2` で
  turn 拒否 → メインの LLM を起動させずコンテキスト消費ゼロ。回答は
  バックグラウンドで生成し、上記いずれかで返す。

## フック仕様の裏取り (claude-plugin-reference hooks.md, v2.1.170)

今日 [確認済] に変わった事実:

- **UserPromptSubmit は exit 2 で turn 拒否できる** [実機検証済 v2.1.170]。
  → 「現在地質問をブロックしてメイン消費ゼロ」は仕組みとして成立。
- **UserPromptSubmit でプロンプト改ざん (書き換え) はできない**。output に
  `updatedPrompt` 系がなく、可能なのは additionalContext 注入か exit 2 の
  丸ごと拒否の二択。input rewrite は PreToolUse の `updatedInput` 限定。
  → 当初案 (「答えるな」と注入して改ざん) は不可。丸ごと deny する
  最新案が仕様に合致。
- **UserPromptSubmit フックは 30s で強制 short-circuit** [spec]。
  → ブロック判定 (同期・軽量) と回答生成 (非同期・バックグラウンド) の
  分離は、設計の好みではなく仕様上の要請。

## YAGNI / 段階導入

構想は当初「常駐デーモン + セッション毎ヘッドレスエージェント + inotify +
DB + アクセススキル」だったが、探索の末に「フックで turn 拒否 + 現在地は
ブラウザ常時表示 + 差し込みは既存 cmux-msg 機構」へ簡素化された。
それでも全部を最初から作るのは過剰。以下の順で、各段が役立つと確認して
から次へ進む:

1. **記録層を手動で回す** — 追跡セッションもインフラもなし。振り返り時
   に未着地決定を相つきで挙げ、`decisions-log` に手で転記。索引が
   「数日後のこれどうなってた?」を実際に減らすか検証。
1. **副作用ゼロの自動確認** — 振り返り時に未確認行へ ls/grep を裏で当て、
   結果を通知で遅延反映 (からうち防止に、確認はユーザ判断時のみ同期で
   取らず2段化)。
1. **追跡セッション + 差分 CSA** — 連続記録の自動化。§6 の未検証を潰して
   から。
1. **フック turn 拒否 + 現在地応答** — メインからの現在地応答剥がし。
1. **ブラウザ可視化** — 静的ビューから。リッチ化は最後。

第一段が無価値なら以降は土台がない。記録層が回り始めるまで上位段は作らない。

## §7: プラグイン名 — `nandakke` (確定: 2026-06-17)

確定: **`nandakke`** (= 「何だっけ?」)。

本ツールの本質「人間が無意識に持っている『あれってどこに何があったっけ?』
の感覚を AI に持たせる」に最も直接的に寄った和語。リポ名は
`claude-nandakke`、プラグイン名は `nandakke`、marketplace 名も `nandakke`。

### 不採用候補

`claude-` prefix 前提 (= registry 競合はほぼ問題にならない)。検討した候補:

- `claude-gist` — 全体のぼんやりした要旨。意味は近いが英単語で抽象度が
  高く、本ツールの「ユーザ視点での体験 (= ど忘れの瞬間)」が伝わらない。
- `claude-atari` (当たり) — 「索引で当たりを付ける」機能名。短く和語で
  良いが、機能の説明に寄りすぎて本質 (= ど忘れを補う) からはやや遠い。
- `claude-index` — 直球だが一般語すぎる。

点 (idea) や線 (flow) に寄せた名前 (idea-mapper / flow-mapper) や、
可視化を含意する語 (atlas / mapper / knowledge-graph) は、コアが索引で
ありグラフ可視化でないという定義からするとややずれるため不採用。
