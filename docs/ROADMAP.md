# ロードマップ

DR-0001 §YAGNI / 段階導入の写し。**各段が役立つと確認してから次へ進む**。
第一段が無価値なら以降は土台がない。記録層が回り始めるまで上位段は作らない。

## 短期 (= 直近着手候補)

### Phase 1: 記録層を手動で回す

- 追跡セッションもインフラもなし。振り返り時に未着地決定を相つきで挙げ、
  `decisions-log` (= 中央記録の試作) に手で転記
- 索引スキーマ 6 列 (topic / 相 / 確信 / 最終言及 / 参照 / next) を実運用
- **検証ポイント**: 索引が「数日後のこれどうなってた?」を実際に減らすか
- 効かなければ以降は土台がない (= ROADMAP の上位段に進まない)

## 中期 (= 構想中)

### Phase 2: 副作用ゼロの自動確認

- 振り返り時に未確認行へ ls/grep を裏で当て、結果を通知で遅延反映
- からうち防止に「確認はユーザ判断時のみ同期で取らず2段化」

### Phase 3: 追跡セッション + 差分 CSA

- 連続記録の自動化
- DR-0001 §6 (未検証) を潰してから着手:
  - CSA の messageid 以降指定の実効性
  - 追跡セッションの文脈寿命と確定相の要約落としサイクル
  - 差分供給のトリガーとコスト

## 長期 / アイデア (= 検討初期)

### Phase 4: フック turn 拒否 + 現在地応答

- UserPromptSubmit で現在地質問を `exit 2` で turn 拒否 → メイン消費ゼロ
- 回答経路: ブラウザ表示 or cmux-msg 経由のソケット差し込み (新規通信
  インフラは作らない)

### Phase 5: ブラウザ可視化

- 静的ビューから。リッチ化 (時系列軸 / アイデア軸 / グラフ) は最後
- Understand-Anything 的なリッチ可視化に引っ張られないこと
- コアは索引、可視化は「あれば嬉しい」止まり (DR-0001)
