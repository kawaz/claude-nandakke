# nandakke 設計

> [English](./DESIGN.md) | 日本語

実装前 (DR-0001 §YAGNI 段階導入)。本ファイルは現時点で「設計の出発点 = DR-0001」を
要約し、実装が進むに従って「現実装の説明」に育てる。

## ドメイン

**AI に、人間が持っている「ぼんやりした正確な全体インデックス」を持たせる。**

人間 (kawaz) は、各プロジェクトで何をどの辺で話し・決め・作ったかを、
正確ではないが「ぼんやり全体」として記憶している (= 索引がある)。だから
「探せば出てくる」と確信して探せる。AI にはこの索引がない。AI が全体を
把握するには docs とコードを全部読むしかなく、重いから端折る → 読まない →
想像で書く事故が起きる。

nandakke は「全部読む (重い) か 読まない (雑) か」の二択を、人間が普段
やっている「ぼんやり全体を把握 → 必要なら正確に取りに行く」に置き換える。

### スコープ

- **対象**: 各プロジェクトのプロジェクト知識 (DR / findings / journal /
  docs) と、それを生むセッションの会話・指示・アイデア・作業
- **対象外**: ルール群 (claude-rules-personal の for-all/for-me)。
  ルール = 規範と本ツールのプロジェクト知識 = 記録は混ぜない
- 横断するとしても「複数プロジェクトのプロジェクト知識」まで

詳細は [DR-0001 §スコープ境界](./decisions/DR-0001-knowledge-index-sidecar-architecture.md#スコープ境界)。

## アーキテクチャ

```
メインセッション (長時間・序盤揮発)
  │ 差分供給 (CSA messageid 以降指定 — 未検証)
  ▼
追跡セッション (1 コンテキスト継続・流れを保持・相と分岐を判断)
  │ 確定した相を書き出し、自身の文脈からは手放す (要約落とし)
  ▼
中央記録層 (永続・横断検索可能)
  ▲
  │ 振り返り (pull・任意のタイミング)
メインセッション ← スキル or ブラウザ表示
```

3 つの役割は単一の核「AI 用の外部索引」の異なる面:

- **記録 (write)**: 会話・指示・アイデア・作業・ツール利用の流れを継続把握し、
  決定とその相、分岐を中央の索引に永続化
- **自己点検 (verify)**: 索引の未確認 (相が推定どまり) を、副作用ゼロの
  確認 (リポの ls/grep) で事実化。重い確認 (他セッションへの問い合わせ) は承認制
- **応答 (serve)**: AI がセッション開始時に索引だけ読んで全体を掴み、詳細は
  索引から辿って必要分だけ正確に読む。現在地参照にも答える

### コアは索引であって可視化ではない

「有機的グラフ」「時系列軸/アイデア軸のビュー」「ブラウザ可視化」は**非コア**。
コアは AI が当たりを付けるための軽い索引。可視化は「あれば嬉しい」止まりで、
最初に作るものではない。

### 索引スキーマ (6 列)

| 列 | 内容 |
|---|---|
| topic | 決定・議論の主題 |
| 相 | spoken / recorded / implemented |
| 確信 | 事実 (機械確認済) / 推定 (追跡セッションの判断) |
| 最終言及 | 最後に触れた時点 (実装では messageid/timestamp) |
| 参照 | セッション uri 等のポインタ (内容は重複させない) |
| next | 次に進めるべき相、または確認手段 (例: `[確認] ls docs/issue/`) |

`確信` 列は必須。推定と事実を混同して記録すると、後で振り返る人間が嘘の
地図を見る。相を倒すのは事実が取れた時だけ。
